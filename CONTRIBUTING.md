# Contributing to the Agentic Delivery Model

The ADM is an open methodology. Contributions are welcome and encouraged.

## How to contribute

### Reporting gaps or issues

If you spot a gap, contradiction or area that needs development, open an issue. Describe the problem clearly and, if possible, suggest an approach. Real-world experience reports are particularly valuable.

### Proposing changes

1. Fork the repository
2. Create a branch for your change
3. Edit MODEL.md (core methodology) or ADOPTION.md (adoption guidance) as appropriate
4. Submit a pull request with a clear description of what you changed and why

### What makes a good contribution

- **Practical experience.** If you have tried implementing any part of the ADM in a real team, your observations are extremely valuable, whether the model worked or not.
- **Regulated environment perspective.** The ADM is designed for enterprise and regulated contexts. Input from people working under PRA/FCA, SOX, HIPAA or similar regimes is particularly welcome.
- **Leadership and governance perspective.** If you have views on how the measurement framework maps to organisational reporting, that input matters.
- **Tooling-specific implementation notes.** The ADM is deliberately tool-agnostic, but practical implementation guides for specific tools (Claude Code, Copilot, Cursor, ADO, Jira) as companion documents would be valuable.

### What to avoid

- Vendor-specific promotion. The ADM is tool-agnostic by design.
- Theoretical arguments without practical grounding. The goal is an implementable model, not an academic paper.
- Changes that assume a specific organisational size. The model should scale from a single product team to a portfolio.
